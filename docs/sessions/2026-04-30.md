# Session — 2026-04-30 (initial app)

Inaugural session. Project went from empty directory to v0.1.0 release.

## Outcomes shipped

- PySide6 main window with cv2 playback, drag-drop loading, FPS readout.
- Draggable + resizable red ROI overlay (8-handle resize).
- Rotation `-180° … +180°` (0.5° step) plus H/V flip; ROI clears on
  orientation change.
- Frame-interval auto-record: `開始記錄` plays the video and OCRs every N
  frames, auto-stops at end-of-video.
- `清除紀錄` with confirmation; `輸出 CSV` (UTF-8 BOM).
- OCR pipeline: rapidocr-onnxruntime in **rec-only mode** — skips DBNet
  detection because the user always supplies a tight ROI. ~30 ms/job vs
  ~870 ms with default det+rec.
- Light upscale-only preprocess (`short_side < 64 → cubic upscale`); CLAHE
  / Otsu / adaptive-threshold removed (empirically harmful on 7-seg LCDs).

## Decisions

| Decision | Why |
|---|---|
| `cv2.VideoCapture` + `QTimer`, not `QMediaPlayer` | Need pixel-level access for OCR cropping. |
| RapidOCR over EasyOCR / PaddleOCR | No torch dependency, ~150 MB install vs ~1.5 GB. |
| Rec-only mode default | Tight ROIs starve DBNet; bypass yields >30 % accuracy lift. |
| Frame-interval recording (not time-interval) | Reproducible across re-runs; aligns with how OCR is sampled. |
| `enforce_admins=false` on branch protection | Owner direct-pushes for solo work; non-owners go through PR. |

## Dispatch team (when OCR was failing on user's tight ROI)

Three parallel agents:
1. **API researcher** — explored `rapidocr-onnxruntime` source; confirmed
   `use_det=False` signature, return shapes, model-swap path, and that
   there is no built-in allowlist (regex postfilter required).
2. **Best-practices researcher** — recommended keeping RapidOCR with
   better preprocessing + ssocr fallback if needed; surveyed competing
   libraries and rejected them on size/torch grounds.
3. **Empirical sweep** — extracted frames from `examples/影片辨識測試影片.mp4`,
   ran 14 preprocess/OCR variants on the actual LCD ROI, confirmed
   rec-only as winner. Surprising side finding: the user's ~9° rotation
   wasn't needed for this video — the LCD was already near-horizontal.

Full write-up: `docs/ocr-investigation.md`.
